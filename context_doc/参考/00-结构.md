# Agent 的骨架

不是砖的清单，而是骨头——撑起整栋大厦的那几根力线。

---

## 一、最小骨架：闭合环

所有 Agent 拆到最后，剩下同一个东西：

```
用户意图
   ↓
┌──────────────────┐
│                  │
│  LLM ──决策──→ 工具  │
│   ↑              ↓  │
│   └── 消息 ←── 结果  │
│                  │
└──────────────────┘
   ↓
最终输出
```

**这是一个闭合的反馈环，不是一条开放的链。**

聊天机器人是链：用户 → LLM → 回复，结束。
Agent 是环：用户 → LLM → 工具 → 结果 → LLM → 工具 → … → 回复。

环和链的区别是本质性的。链是一次性的信息传递；环是持续的感知-决策-行动循环。李继刚说"链是射出去的箭，环才能形成积淀"——Agent 的结构验证了这一点。

**证据：所有项目的核心都是同一个环**
- Codex: `run_turn()` 循环，needs_follow_up → 继续
- Pi-mono: `AgentSession` 事件队列，agent_end → 检查是否需要下一轮
- Learn CC s01: `while tool_use: execute → loop`，41 行
- AI Agents: ReAct = Thought → Action → Observation → Thought...

形态不同（Rust 状态机、Promise 链、Python while），但骨头是同一根。

## 二、六根骨头

拆解所有项目后，Agent 的骨架由六根骨头构成。不是六个"功能模块"——是六个**结构性的力承载点**，去掉任何一个，上面的东西就塌了。

### 骨头 1：环（控制流）

环本身。没有环就没有 Agent。

```python
# 最小形态（Learn CC s01, 41行）
while True:
    response = LLM(messages, tools)
    if tool_use: execute(); continue
    else: break
```

环的本质是**LLM的输出经过世界的变换后再次成为LLM的输入**。这不是普通的循环——是一个反馈控制系统，LLM是控制器，工具是执行器，消息是反馈通道。

生产级的环比教学版复杂在哪里？**不是环变了，而是环上长出了控制机构**：
- Codex: CancellationToken（中断环）、Turn-Scoped Context（每圈独立的上下文）
- Pi-mono: Promise链串行化（保证环的每一步顺序执行）、steering messages（从外部注入中断信号）
- 共同点：环的形状没变，但增加了**启动、中断、恢复、超时**四种控制

### 骨头 2：消息（状态载体）

消息数组不是"聊天记录"。**它是 Agent 的全部工作记忆。**

```
SystemMessage  → 定义 Agent 是谁、能做什么（身份）
HumanMessage   → 用户的意图（输入）
AIMessage      → LLM 的决策和推理（思考）
ToolMessage    → 世界的反馈（感知）
```

所有东西都通过消息流动。Agent 没有"变量"、没有"内部状态"——消息数组就是它的意识流。这解释了为什么：
- **上下文压缩 = 有选择的遗忘**（不是优化，是记忆管理）
- **子 Agent = 全新的意识**（空消息数组 = 干净的工作记忆）
- **技能注入 = 临时植入知识**（往消息里塞一段 tool_result）
- **持久化 = 保存意识状态**（序列化消息数组）

Pi-mono 把消息做成树结构（id/parentId），不是为了花哨——是因为意识可以分叉。你可以从任何一个决策点 fork 出一条新的思维线。

### 骨头 3：工具（行动边界）

工具不是"给 Agent 增加能力"。**工具是约束。**

没有工具定义的 LLM 可以说任何话、承诺任何事，但什么都做不了。工具定义的 JSON Schema 是一个约束：你只能从这些动作中选择，每个动作的参数必须符合这个结构。

**约束产生可能性空间。** 这和李继刚说的"结构是约束构成的可能性空间"完全一致：
- 6 个工具（Read/Write/Edit/Bash/Glob/Grep）定义了一个"文件系统操作"的可能性空间
- 20+ 个工具（Codex）定义了一个更大的可能性空间
- 工具越多不是越好——是可能性空间的形状不同

工具的内部结构也是统一的：

```
输入侧：JSON Schema（结构化的意图描述）
   ↓
执行侧：handler 函数（意图变成行动）
   ↓
输出侧：文本结果（世界的反馈）
```

所有项目——Codex 的 Rust trait、Pi 的 AgentTool 接口、Learn CC 的 dict dispatch——形态不同，但骨头相同：Schema → Execute → Result。

### 骨头 4：上下文窗口（头骨）

上下文窗口是 Agent 的**硬物理约束**，就像头骨限制了大脑的大小。

如果上下文无限大，Agent 架构会简单得多——不需要压缩、不需要子 Agent、不需要外部状态、不需要按需加载。所有复杂性的根源是：**记忆是有限的。**

每个项目对这个约束的回应：

| 策略 | 本质 | 谁在用 |
|------|------|--------|
| 消息移除 | 截肢（粗暴但快） | Yinxi |
| LLM 摘要 | 提炼记忆（保留结构，丢掉细节） | Pi-mono, Learn CC, Codex |
| 子 Agent | 开辟新脑（隔离上下文） | Learn CC s04, Codex |
| 外部状态 | 外置硬盘（任务图、文件） | Learn CC s07 |
| 按需加载 | 用时才装入（技能、引用） | Claude Code Skills, Learn CC s05 |

注意这些策略的优先级顺序：先学会遗忘，再学会分身，再学会外存，最后学会按需。这不是随机的——是从最紧迫的痛到最精细的解法。

### 骨头 5：权限膜

在 LLM 决策和工具执行之间，有一层膜。这层膜决定：**这个行动是否被允许执行？**

```
LLM 决策："我要执行 rm -rf /"
        ↓
   ┌────────────┐
   │  权 限 膜   │ ← 审批？沙箱？Hook？
   └────────────┘
        ↓
   允许 / 拒绝 / 修改 / 降级
```

这根骨头在教学项目中**完全缺失**（Learn CC、AI Agents From Scratch 里没有任何审批机制）。但在生产级项目中，它是最重要的结构之一：

- **Codex**: 三步审批管线（检查 → 请求 → 决定）+ 沙箱策略（文件系统/网络/Windows容器三层）
- **Claude Code**: 四层权限（组织强制 → 用户设置 → 命令级 → 默认）
- **Pi-mono**: 扩展 Hook 包装工具（before/after，可修改参数、跳过执行、替换结果）

权限膜不是"安全功能"——**它是 Agent 和自动化脚本的分界线**。没有膜的 Agent 就是一个会说话的 cron job。有了膜，人类在关键节点保持控制权，Agent 在安全边界内自主行动。

### 骨头 6：系统提示（基因）

系统提示定义 Agent **是谁、能做什么、边界在哪里**。它是 Agent 的基因——在环开始转之前就决定了 Agent 的形状。

系统提示不是一段静态文字。所有项目的系统提示都是**动态构建**的：

```
固定部分（身份、行为准则、输出格式）
  +
环境信息（当前目录、操作系统、时间）
  +
工具描述（从工具注册表动态生成）
  +
项目上下文（CLAUDE.md / AGENTS.md / YINXI.md）
  +
技能元数据（名称和触发描述，按需展开）
```

Claude Code 的渐进式加载是这根骨头最精细的实现：
- Level 1: 技能元数据始终在系统提示中（~100 tokens/skill，用于触发检测）
- Level 2: 匹配到关键词后加载 SKILL.md 正文
- Level 3: Claude 判断需要时加载 references/scripts

这不是"加载优化"——这是**注意力的约束与自由**。约束了默认加载量，换来了按需展开的自由。

## 三、力的传导：意图如何变成行动

六根骨头不是并列的。它们之间有力的传导关系：

```
系统提示(基因)
    │
    │ 定义能力边界
    ↓
用户意图 → 消息(状态载体) → LLM 决策
                                │
                                │ 工具调用请求
                                ↓
                           权限膜(过滤)
                                │
                                │ 允许/拒绝/修改
                                ↓
                           工具(行动边界)
                                │
                                │ 执行结果
                                ↓
                           消息(状态载体) → LLM 再次决策
                                │
                                │ 如果上下文满了
                                ↓
                           上下文窗口(头骨) → 压缩/分身/外存
                                │
                                │ 继续环
                                ↓
                              ...
```

力线是单向的：**意图 → 决策 → 过滤 → 行动 → 反馈 → 决策**。但整体是一个环。

每一个生产级的复杂度都是在这条力线上的某个点加了一个**变压器**：
- 并行工具执行：在"行动"这个点分叉，多条力线同时走
- 子 Agent：在"决策"这个点分裂，开一个新的环
- 事件系统：在力线的每个节点挂监听器，让外部代码能观察和介入
- 扩展/Hook：在力线的关键节点插入拦截器

## 四、生长的逻辑：从 41 行到生产级

Learn Claude Code 的 12 课揭示了 Agent 的生长顺序。这个顺序不是教学设计——它反映了**结构性的优先级**：

```
第一阶段：能活着（s01-s02）
├── s01: 环 ← 没有环就不是 Agent
└── s02: 工具分发 ← 没有手脚就不能行动

第二阶段：能思考（s03-s05）
├── s03: 计划/Todo ← 没有计划就漫游
├── s04: 子 Agent ← 没有分身就被复杂度淹没
└── s05: 技能注入 ← 没有知识就瞎做

第三阶段：能记忆（s06-s07）
├── s06: 上下文压缩 ← 不会遗忘就爆掉
└── s07: 持久化状态 ← 不会外存就丢失

第四阶段：能协作（s08-s12）
├── s08: 后台任务 ← 串行太慢
├── s09: 消息总线 ← 多个 Agent 需要通信
├── s10: 请求-响应协议 ← 通信需要规范
├── s11: 自治轮询 ← Agent 需要主动性
└── s12: 工作树隔离 ← 并行需要安全边界
```

**每一层生长都是对一个矛盾的回应：**

| 矛盾 | 回应 | 本质 |
|------|------|------|
| 不能行动 | 工具系统 | 给意图一个出口 |
| 行动没方向 | 计划系统 | 给环一个目标 |
| 单脑不够用 | 子 Agent | 分裂意识 |
| 知识太多装不下 | 按需加载 | 懒加载 |
| 记忆会满 | 压缩 | 有选择的遗忘 |
| 重启就失忆 | 持久化 | 意识的序列化 |
| 一次只做一件事 | 后台/并行 | 时间的并发 |
| 一个人干不完 | 团队/协议 | 社会化 |
| 等人指令太被动 | 自治轮询 | 主动性 |
| 并行会冲突 | 工作树隔离 | 空间的隔离 |

注意这个顺序：**先活着，再思考，再记忆，最后协作。** 这和生物进化的顺序惊人地一致——单细胞 → 神经系统 → 记忆 → 社会性。

## 五、三个深层洞见

### 洞见 1：Agent 的本质是"有限算力在无限可能性空间中的搜索"

李继刚在播客中反复提到"有限算力"。Agent 的结构完美体现了这一点：

- 上下文窗口是有限的（算力的硬约束）
- 工具集是有限的（行动空间的约束）
- 每一轮只能做有限的决策

所有的架构设计——压缩、分身、外存、按需加载——都是在**有限算力下最大化搜索效率**的策略。这和他说的"人的算力是有限的，从信息中取出的结构因算力而变"是同一件事。

### 洞见 2：消息数组是 Agent 的"取景框"

李继刚说"取景框决定你看到什么"。对 Agent 来说，消息数组就是它的取景框：

- 系统提示决定了取景框的角度（你是谁）
- 对话历史决定了取景框的焦点（你在看什么）
- 压缩策略决定了取景框的景深（你能看多远）
- 技能注入决定了取景框的分辨率（你能看多细）

两个拥有相同模型但不同消息数组的 Agent，面对同一个任务会做出完全不同的决策。**不是模型不同，是取景框不同。**

### 洞见 3："干"和"湿"的分界线就是工具边界

李继刚把世界分成"干"（可形式化、可计算）和"湿"（感性、直觉、意志）。

在 Agent 架构中，这条分界线就是工具的 JSON Schema：
- Schema 能描述的 = 干的（参数类型、取值范围、结构约束）
- Schema 描述不了的 = 湿的（审美判断、意图理解、价值选择）

LLM 在"湿"的一侧：它理解模糊的意图，做出审美判断，选择行动方向。
工具在"干"的一侧：它接受结构化输入，执行确定性操作，返回客观结果。

**Agent 的结构就是在干湿之间搭建桥梁。** 消息是桥面，工具 Schema 是桥墩，权限膜是护栏。
